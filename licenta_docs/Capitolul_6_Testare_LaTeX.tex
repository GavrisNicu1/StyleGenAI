\chapter{Testarea și Validarea Sistemului StyleGenAI}
\label{cap:testare}

Acest capitol validează funcționalitățile principale ale aplicației StyleGenAI în raport cu implementarea curentă (MVP). Accentul este pus pe trei direcții: corectitudine funcțională, comportament la erori și timp de răspuns. În plus, sunt delimitate explicit funcționalitățile implementate de cele planificate.

\section{Strategia de testare}
Testarea a combinat:
\begin{itemize}
    \item teste funcționale end-to-end pentru autentificare, generare și istoric;
    \item teste de integrare API pentru comunicarea frontend-backend;
    \item teste de robustețe pentru input invalid și token lipsă/invalid;
    \item măsurarea timpilor de răspuns pentru endpoint-urile principale.
\end{itemize}

\subsection{Niveluri de testare utilizate}
În cadrul MVP-ului, strategia a fost împărțită pe niveluri, pentru a reduce riscul de regresie funcțională:
\begin{itemize}
    \item \textbf{nivel endpoint}: validarea codurilor de răspuns HTTP și a structurii JSON;
    \item \textbf{nivel flux}: verificarea secvențelor complete (login $\rightarrow$ generare $\rightarrow$ salvare $\rightarrow$ istoric);
    \item \textbf{nivel UX}: verificarea mesajelor de eroare și a continuității interacțiunii în ecranele principale.
\end{itemize}

Această împărțire este utilă într-un proiect în care backend-ul gestionează atât validări de date, cât și logică de recomandare, iar frontend-ul are responsabilitatea de a interpreta corect răspunsurile.

\subsection{Criterii de acceptanță}
Pentru fiecare scenariu, rezultatul a fost considerat valid doar dacă au fost îndeplinite simultan următoarele condiții:
\begin{itemize}
    \item codul HTTP este conform așteptărilor pentru scenariul testat;
    \item payload-ul conține câmpurile obligatorii pentru ecranul consumator;
    \item aplicația nu intră în stare blocată, iar utilizatorul poate continua fluxul;
    \item mesajul de eroare este explicit în caz de eșec.
\end{itemize}

\section{Mediul de testare}
Testele au fost realizate în mediul de dezvoltare:
\begin{itemize}
    \item \textbf{Frontend}: React Native (Expo), rulare Web și mobil (Expo Go);
    \item \textbf{Backend}: Flask, rulare locală pe \texttt{http://localhost:5000};
    \item \textbf{Bază de date}: SQLite (implicit) și suport SQL Server prin stratul \texttt{database.py};
    \item \textbf{Resurse test}: imagini JPG/PNG/WebP valide și fișiere deliberate invalide.
\end{itemize}

\subsection{Set de date și variații folosite}
Pentru a evita concluzii bazate pe un singur tip de intrare, testele au inclus:
\begin{itemize}
    \item imagini cu fundal simplu și imagini cu fundal complex;
    \item piese vestimentare cu palete cromatice apropiate și palete cu contrast ridicat;
    \item combinații complete pe categorii și combinații insuficiente (pentru validarea precondițiilor de generare);
    \item dimensiuni diferite de fișiere, în limitele acceptate de backend.
\end{itemize}

\subsection{Reproductibilitatea testelor}
Execuția testelor a urmărit același protocol la fiecare iterație: pornire backend, autentificare, rulare scenarii pozitive, rulare scenarii negative și colectare timpi endpoint. Această repetabilitate permite compararea versiunilor aplicației pe același set de pași.

\subsection{Protocol operațional de execuție a testelor}
Pentru a asigura consistența rezultatelor, protocolul de rulare a inclus următorii pași:
\begin{enumerate}
    \item inițializarea backend-ului și verificarea endpoint-ului \texttt{/health};
    \item autentificare cu utilizator de test și obținerea token-ului JWT;
    \item rularea fluxului principal de recomandare prin \texttt{/get\_suggestion};
    \item rularea operațiilor de persistență (save/history/like/delete);
    \item rularea scenariilor negative (token lipsă/invalid, payload invalid, tip fișier neacceptat);
    \item notarea răspunsurilor și a timpilor de execuție pentru comparații ulterioare.
\end{enumerate}

Acest protocol a fost utilizat uniform în sesiunile de testare, astfel încât deviațiile observate să poată fi atribuite schimbărilor de cod, nu variațiilor de procedură.

\section{Validarea integrării Frontend-Backend}
Integrarea a verificat corespondența dintre ecrane și endpoint-uri:
\begin{itemize}
    \item ecranul principal trimite \texttt{POST /get\_suggestion} cu \texttt{multipart/form-data} (fișiere, categorii, filtre);
    \item după răspunsul principal, aplicația poate solicita \texttt{POST /web\_outfit} pentru propuneri externe;
    \item ecranele de istoric consumă \texttt{GET /outfits/history}, \texttt{POST /outfits/\{id\}/like}, \texttt{DELETE /outfits/\{id\}};
    \item token-ul JWT este transmis în antetul \texttt{Authorization} pentru rutele protejate.
\end{itemize}

Nu au fost observate incompatibilități majore de format JSON, iar erorile deliberate au fost tratate corect în interfață.

\subsection{Contracte de date (request/response)}
În validarea integrării, accentul a fost pus pe consistența contractelor:
\begin{itemize}
    \item pentru \texttt{/get\_suggestion}, request-ul trebuie să includă liste aliniate \texttt{files} și \texttt{categories};
    \item pentru rutele protejate, antetul \texttt{Authorization} este obligatoriu;
    \item pentru istoricul ținutelor, răspunsurile includ date suficient de complete pentru listare și acțiuni ulterioare (like/delete).
\end{itemize}

Respectarea acestor contracte a redus semnificativ riscul de erori la granița dintre frontend și backend.

\section{Testarea de robustețe și cazuri negative}
Au fost verificate explicit scenarii de eșec:
\begin{itemize}
    \item fișier cu MIME neacceptat la upload;
    \item lipsa câmpurilor obligatorii în request;
    \item token invalid/expirat pentru endpoint-uri securizate;
    \item indisponibilitate temporară backend (simulată) și verificarea mesajelor de eroare în frontend.
\end{itemize}

Rezultatele confirmă validări corecte în backend și un comportament predictibil în frontend.

\subsection{Comportament observat la erori}
În scenariile negative, aplicația menține un comportament stabil:
\begin{itemize}
    \item backend-ul returnează mesaje explicite pentru erorile de validare;
    \item frontend-ul afișează feedback fără a pierde starea critică a fluxului;
    \item după remedierea inputului, utilizatorul poate relua operația fără restartul aplicației.
\end{itemize}

Această proprietate este importantă pentru utilizare reală, unde erorile de input sunt frecvente.

\section{Performanță și timp de răspuns}
Măsurătorile au fost efectuate pe serii scurte de cereri consecutive în mediu local de dezvoltare. Valorile sunt orientative, dar utile pentru validarea UX în MVP.

\begin{itemize}
    \item \texttt{/health}: 2--5 ms (baseline);
    \item endpoint-uri de retrieval (ex. \texttt{/outfits/history}): sub 50 ms;
    \item \texttt{/get\_suggestion}: medie 400--500 ms (procesare imagine + scorare combinații);
    \item \texttt{/web\_outfit}: 700--1000 ms (latente externe mai mari).
\end{itemize}

Sistemul răspunde previzibil: operațiile rapide nu blochează aplicația, iar apelurile mai lente (web) rămân opționale.

\subsection{Indicatori statistici utilizați}
Pentru un eșantion de $N$ cereri, indicatorii utilizați sunt:
$$
\mu = \frac{1}{N}\sum_{i=1}^{N} t_i, \quad
\sigma = \sqrt{\frac{1}{N}\sum_{i=1}^{N}(t_i-\mu)^2}
$$

\subsection{Interpretarea practică a latenței}
Din perspectivă UX, endpoint-urile se pot împărți în două categorii:
\begin{itemize}
    \item \textbf{operații instantanee} (de ordinul zecilor de milisecunde): autentificare, retrieval istoric;
    \item \textbf{operații computaționale} (sute de milisecunde): generarea de recomandări și propunerile web.
\end{itemize}

Pentru categoria a doua, aplicația utilizează feedback vizual de încărcare, astfel încât timpul de așteptare să fie explicit pentru utilizator, fără percepția de blocare.

\subsection{Factori care influențează performanța}
Valorile măsurate depind de mai mulți factori operaționali:
\begin{itemize}
    \item dimensiunea imaginilor încărcate;
    \item complexitatea cromatică a imaginilor;
    \item numărul de piese și, implicit, numărul de combinații evaluate;
    \item latența externă pentru sursele web din \texttt{/web\_outfit}.
\end{itemize}

\section{Validarea logicii de recomandare}
Logica de recomandare a fost validată prin verificarea următoarelor mecanisme implementate efectiv:
\begin{itemize}
    \item extragere culoare dominantă din imagine (pe pixeli non-transparenți);
    \item grupare piese pe categorii;
    \item generare combinații prin produs cartezian;
    \item calcul scor compozit în \texttt{rules.py};
    \item bonus de trend în motorul de generare;
    \item selecția combinației cu scor maxim.
\end{itemize}

Testele manuale au confirmat că modificarea contextului (stil/sezon/gen) influențează rezultatul, iar lipsa inventarului minim produce răspuns de eroare explicit.

\subsection{Verificare pe scenarii comparative}
Pentru același set de imagini, au fost comparate rezultate între configurații diferite de context:
\begin{itemize}
    \item schimbarea sezonului influențează selecția finală prin modificarea scorului;
    \item schimbarea stilului preferat favorizează alte combinații în clasamentul final;
    \item la inventar insuficient, motorul nu forțează rezultat, ci întoarce eroare explicită.
\end{itemize}

Acest comportament confirmă că algoritmul aplică reguli consistente, nu rezultate aleatorii.

\subsection{Determinism și consistență}
În condiții identice de input și context, răspunsul motorului rămâne stabil (aceeași combinație dominantă). Această proprietate este esențială pentru încrederea utilizatorului și pentru depanare, deoarece aceeași cauză produce același efect în testări repetate.

\section{Testarea securității și observații}
\subsection{Controale validate}
\begin{itemize}
    \item validare JWT: token lipsă/invalid/expirat returnează 401;
    \item stocare parole hash-uită cu \texttt{bcrypt};
    \item query-uri parametrizate în stratul de acces la date (protecție de bază la SQL injection);
    \item CORS configurat pe allowlist de origini (development/producție prin variabile de mediu).
\end{itemize}

\subsection{Configurația CORS și impactul în testare}
Configurația CORS curentă aplică o listă de origini permise, cu valori distincte pentru development și producție. În practică, acest lucru produce două efecte utile:
\begin{itemize}
    \item reduce consumul neautorizat al API-ului din contexte browser externe;
    \item păstrează flexibilitatea în dezvoltare locală (origini de tip localhost).
\end{itemize}

În testare, accesul cross-origin din origini nepermise este respins la nivel de browser, ceea ce validează corectitudinea politicii de integrare.

\subsection{Limitări curente}
\begin{itemize}
    \item rate limiting per IP nu este încă implementat în backend;
    \item hardening suplimentar (WAF, throttling avansat, monitorizare de securitate) rămâne etapă de producție extinsă.
\end{itemize}

\section{Limitări metodologice ale testării}
Rezultatele prezentate sunt relevante pentru validarea MVP, dar au limitări care trebuie menționate explicit:
\begin{itemize}
    \item setul de date test este controlat și nu acoperă toate variațiile posibile din utilizarea reală;
    \item mediul local reduce variabilitatea infrastructurii (nu reflectă complet un deployment distribuit);
    \item testarea de încărcare concurentă extinsă nu a fost focusul principal al acestui capitol.
\end{itemize}

În consecință, capitolul validează corect funcționalitatea implementată și comportamentul general de performanță, iar profilarea avansată de producție rămâne pas următor.

\section{Amenințări la validitatea rezultatelor}
Pentru rigurozitate metodologică, amenințările la validitate sunt grupate în trei categorii.

\subsection{Validitate internă}
Riscul principal este influența factorilor necontrolați din mediul local (încărcare CPU, procese concurente), care pot afecta timpii de răspuns. Acest risc a fost redus prin repetarea scenariilor pe același setup.

\subsection{Validitate externă}
Rezultatele din mediu local nu se transferă direct 1:1 către producție, unde intervin factori precum infrastructura de rețea, TLS termination, proxy invers și trafic concurent mai mare.

\subsection{Validitate de construcție}
Metricile folosite în MVP urmăresc funcționalitatea și robustețea operațională, nu evaluarea unui model ML complex. Din acest motiv, concluziile sunt formulate strict în raport cu logica de recomandare implementată curent.

\subsection{Măsuri de reducere a riscurilor}
Pentru reducerea amenințărilor de validitate în iterațiile viitoare, sunt recomandate:
\begin{itemize}
    \item execuția testelor automate în pipeline dedicat;
    \item rularea benchmark-urilor pe infrastructură apropiată de producție;
    \item extinderea eșantioanelor de test pentru imagini și contexte vestimentare.
\end{itemize}

\section{Concluzii}
Testarea confirmă că StyleGenAI este funcțional pentru fluxurile esențiale: autentificare, generare ținută, recomandări alternative și istoric. Sistemul tratează corect erorile uzuale și oferă timpi de răspuns acceptabili pentru un MVP.

Actualizarea CORS pe allowlist îmbunătățește securitatea de integrare între client și API, limitând accesul cross-origin la origini autorizate. Pentru etapele viitoare sunt recomandate implementarea rate limiting-ului, extinderea setului de teste automate și consolidarea observației operaționale în medii de producție.
